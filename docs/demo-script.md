# 演示脚本 - 5 分钟答辩流程

## 演示准备

- 确保已执行 `npm run dev`，应用正常启动
- 应用处于 full 模式
- 准备好讲述每个环节的设计思路

---

## 演示流程

### 第 1 分钟：项目概述与启动

**操作**：
1. 展示应用启动后的完整界面
2. 介绍项目定位：生态圈三号是一个基于 MVU 架构的桌面第三空间

**讲述要点**：
- 技术栈：Electron + Vite + React + TypeScript
- 核心架构：MVU（Model-View-Update）
- 设计理念：为用户提供一个介于虚拟与现实之间的桌面伴侣空间
- 四个角色：米卡、南星、穗穗、林也，各有独特性格

**界面展示**：
- 指出四个角色在生态舱中的位置
- 展示角色卡片（名字、状态、对话）
- 介绍顶部标题栏和模式切换按钮

---

### 第 2 分钟：WorldState 面板与状态管理

**操作**：
1. 指向 WorldState 面板
2. 逐一介绍各字段含义

**讲述要点**：
- `uiMode`：界面模式（mini/full/sleep）
- `owner.mode`：主人当前模式（专注/休息/陪聊/勿扰）
- `weather.kind`：天气状态（晴天/雨天/阴天/高温/降温）
- `focus.min`：今日专注分钟数
- `llm.mode`：LLM 调用模式（安静/正常/活跃）
- `pending`：待处理的社交事件

**设计亮点**：
- 所有状态集中管理，便于调试和理解
- 状态变化通过 WorldEvent 驱动
- reducer 纯函数，易于测试

---

### 第 3 分钟：交互演示 - 主人模式切换

**操作**：
1. 点击"专注"按钮
2. 观察角色反应和 WorldState 变化
3. 点击"休息"按钮
4. 点击"陪聊"按钮
5. 点击"勿扰"按钮

**讲述要点**：
- 每次切换都通过 `OWNER_MODE_CHANGED` 事件驱动
- 角色会根据主人模式调整行为和对话
- 事件日志实时记录所有变化
- 这种设计让状态变化可追溯、可调试

**界面展示**：
- 指出 WorldState 面板中 owner.mode 的实时更新
- 指出事件日志中的新记录
- 指出角色对话的变化

---

### 第 4 分钟：交互演示 - 角色拖拽与社交互动

**操作**：
1. 拖拽米卡靠近南星
2. 观察 pendingSocialEvent 的变化
3. 拖拽穗穗靠近林也

**讲述要点**：
- 角色拖拽通过 `CHARACTER_DRAGGED` 事件实现
- 角色靠近触发 `CHARACTER_NEAR` 事件
- 系统自动生成 pendingSocialEvent（pair_chat 类型）
- 未来可扩展为实际的社交互动逻辑

**设计亮点**：
- 社交事件是延迟处理的，不会立即打断当前状态
- 事件有种子文本，便于 LLM 生成后续对话
- 关系亲密度会随互动增加

---

### 第 5 分钟：技术架构总结与未来展望

**操作**：
1. 切换到 mini 模式，展示界面简化
2. 切换回 full 模式
3. 展示事件日志的完整历史

**讲述要点**：

**架构优势**：
- MVU 架构让状态管理清晰可预测
- 所有状态变化通过 dispatch -> reducer -> new state 流程
- 事件驱动设计便于扩展新功能
- 纯函数 reducer 易于单元测试

**模块化设计**：
- `shared/types.ts`：类型定义
- `shared/events.ts`：事件类型
- `shared/reducer.ts`：状态更新逻辑
- `renderer/`：UI 展示层
- `main/`：Electron 主进程

**天气规则系统**：
- 5 种天气类型，各有独立规则
- 每种天气包含 moodHint、actionBias、sampleEvents
- 示例事件温和、生活化、正向中性

**未来展望**：
- 接入真实天气 API
- 实现 LLM 驱动的角色对话
- 添加记忆系统和日报生成
- 支持更多社交互动类型

---

## 演示技巧

1. **节奏控制**：每个环节约 1 分钟，不要超时
2. **重点突出**：强调 MVU 架构和事件驱动设计
3. **互动性**：多做操作演示，少讲理论
4. **准备问答**：预想评委可能问的问题
   - 为什么选择 MVU 而不是 MVC？
   - 如何保证状态的一致性？
   - 未来如何扩展新的事件类型？
